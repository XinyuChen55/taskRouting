1背景与目标

公司有一个内部基座模型,同时也可能接触到其它候选模型(尺寸/版本不同,或外部模型)。业务每天会涌入各种任务:从财报里抽数字、给客服对话分类意图、总结研报、回答投研问题……

不同模型在不同任务上强弱差异很大,而且越强的模型越贵、越慢。我们想要一个机制:

  • 评测(离线):用一套固定标准,量出"每个模型在每类任务上有多强",产出一张能力矩阵。
  • 路由(在线):新任务进来时,自动判断它属于哪类,选出"能达标里最便宜"的模型。

最大的约束:真实用例数量有限,无法靠大规模标注训练一个复杂路由器。本课题的全部设计都围绕"小数据下也成立"展开。

这件事的价值:抽取、分类这类高频任务,大多数用便宜模型就能做好;只有少数复杂任务才需要贵模型。把它们分开路由,本身就能显著省钱 —— 这是项目的直接 ROI。

2四条核心设计原则

这四条决定了整个方案的形态,务必先理解再开工。

① 把"评测"和"路由"分开做

它们是两件事:评测是季度/版本更新时跑一次,产出"模型×任务类"打分表;路由是实时的,输入是新任务特征 + 那张表。评测是路由的输入。先把评测做扎实,路由初期就是一张查找表。

② 按任务"类别"路由,不按单个任务 —— 这是破解小数据的关键

单个任务级别的路由需要海量标注。但如果把用例聚成 5–10 个任务类(抽取 / 分类 / 摘要 / 问答…),你只需要刻画每个类,数据需求从"成百上千"降到"每类 20–50 条"。新任务来了先分类(比判断难度容易),再查"这个类哪个模型最优"。

③ 能力是多维的,不要压成一个分数

一个排行榜数字会掩盖掉最关键的"按任务变化的强弱"。我们要的是矩阵 [模型 × 任务类 → 质量分 + 通过率 + 时延 + 成本],而不是"模型 A 总分 87"。

④ 金融场景有些维度是"硬门槛",不能进平均分

比如"编造了一个财务数字"。一个文笔流畅但把净利润数字编错的模型,在抽取类里应当直接判不合格,而不是靠流畅度把平均分拉回来。硬门槛是 pass/fail,凌驾于质量分之上。

3任务分类体系(本课题的核心产物)

下面是初版 taxonomy,你要用公司真实用例去校验、补充、合并。每类都给了具体例子和"能区分模型强弱的难例"——这些难例就是你 golden set 里最该收的样本。

类 1 · 结构化抽取(表格 / 财务数据)

从财报 PDF 抽三大报表成 JSON;从研报抽"目标价/评级/盈利预测"成表;从减持公告抽"股东/比例/价格/时间窗"。

简单 CASE(便宜模型够)
输入:一张干净的单页利润表
输出:{"营业收入": 1234.5, "归母净利润": 89.0, "单位": "万元"}
能拉开模型差距的难例 ← 重点收集
  • 跨页续表:表格断在两页,下半页没表头 → 弱模型丢行/错位
  • 合并单元格 / 多级表头:三列对齐错位
  • 口径陷阱:合并 vs 母公司、调整前 vs 调整后(脚注)抓错列
  • 单位量纲:万元/亿元/元混用,数字对但量级错
  • 科目同名消歧:不同表里同名科目含义不同

建议把本类拆成两个子类:抽取·规整表(便宜)与 抽取·复杂版式(强),分别路由。

类 2 · 意图识别 / 分类

客服意图(查净值/赎回/投诉/产品咨询/风险测评/转人工);公告类型;研报观点情感(看多/看空/中性);合规风险分级。

简单 CASE(便宜模型完胜,别用贵模型烧钱)
"我想把上个月买的那个基金赎回一部分" → 赎回申购
难例
  • 多标签:"收益这么差,我要投诉并赎回" → {投诉, 赎回}
  • 边界模糊:"这个产品还能买吗" = 咨询 还是 申购意图?
  • 反讽/否定:"这票还看多?早该清了" → 实为看空
  • 长尾高代价类:占比 1% 但漏判代价极高(投诉、合规高危)

类 3 · 长文摘要 / 要点提取

2 万字研报 → 投资逻辑 3 点 + 风险 2 点;业绩说明会纪要 → 关键边际变化。考验长上下文 + 抓重点 + 不脑补原文没有的数字/结论。

类 4 · 文档问答 / 多步推理

"这家公司 Q1 毛利率为什么下滑?"(跨章节综合);"三家券商的分歧在哪?"(多文档对照);"这份财报触发我们风控哪几条?"(推理+规则匹配)。这是强模型主场,不该为省钱降级。

类 5 · 改写 / 合规化 / 翻译

内部观点改成合规话术(删"必涨""稳赚");财经中英互译;对外脱敏。门槛是合规而非质量,可配规则后置校验。

类 6 · 数据规整 / 实体消歧

简称→统一代码;日期/单位归一;实体消歧(例:001259 是利仁科技,不是国轩科技 —— 这种 LLM 凭记忆会答错)。便宜模型 + 词典兜底即可。

4评分卡:rubric 与硬门槛

每个任务类要配一张评分卡 = 几条质量 rubric(打分用) + 几条硬门槛(pass/fail)。示例:

任务类质量 rubric(举例)硬门槛(不过出局)
抽取·复杂版式覆盖率、字段正确率、结构对齐每个数字可逐字溯源、量纲正确、不补造科目
意图分类整体准确率、边界一致性"投诉/合规高危"召回率 ≥ 阈值
长文摘要要点命中、精炼度不引入原文没有的数字/论断
文档问答正确性、推理链合理结论有原文接地,不臆测
改写合规语义保真、可读性合规红线词 / 绝对化表述零容忍
硬门槛的意义:金融场景里"编一个看起来合理的数字"比"漏一个数字"危险得多。硬门槛让这类错误一票否决,而不是被流畅度平均掉。每个任务类至少定义 1 条数字/事实忠实度门槛。

5评测方法论(专为小数据设计)

  1. 先写 rubric,再跑模型。不用调任何模型,却是最高杠杆的一步。
  2. 用成对比较(pairwise),不用绝对打分。同样样本量,成对比较榨出的信号更多、更稳;多模型排序用 Elo / Bradley–Terry 聚合。绝对分主观漂移大。
  3. LLM-as-judge + 人工校准。抽 30–50 条让人也判一遍,算评委与人的一致性(Cohen's κ);κ 不达标就改 rubric 或换评委。判数字对错的评委必须能调用计算/检索,不能凭语感。
  4. 刻意收集难例和失败例。平均分会被高频简单任务稀释,决定路由的是边界。主动把模型翻车的 case 留进 golden set。
  5. 统计审慎(小 N 必踩):你会看到一堆"A 比 B 高 2 分",大半是噪声。对成对结果做 bootstrap 置信区间;差距没拉开就判平局、选更便宜的那个。
  6. 记录成本与时延,和质量分并列进矩阵。

产出 = 能力矩阵:[模型 × 任务类 → 质量分 + 硬门槛通过率 + p95 时延 + 单位成本]。

6路由器的演进路线 —— 别一上来就训分类器

数据量路由器形态说明
几十条决策表 / 规则任务分类 → 查能力矩阵 → 选"过门槛里最便宜的"。可解释、可审计。
几百条kNN / 向量检索路由embedding 找最相似历史任务,看当时谁赢。抗过拟合、可解释。
上千条+训练轻量分类器 / 在线学习才考虑按单任务(而非按类)路由。
注意:早期用学到的路由器在小数据上必然过拟合。规则表更稳、更可审计(金融还要可解释)。两个加分项:
① 质量×成本前沿:路由不是选最强,是选"过质量线里最省的"。
② 级联(cascade):先上便宜模型,门槛不过或自评置信度低再升级强模型。
线上每次决策都记日志(任务→选了谁→结果)——这就是后续涨数据的来源,路由系统自己产训练数据。

7你要交付什么(Deliverables)

D1
任务分类体系文档 —— 用真实用例校验后的 5–10 个任务类:每类定义 + 3–5 条 rubric + 硬门槛 + 难例模板。(本文档第 3–4 节是初版,你要落地确认)
D2
Golden set —— 每类 20–50 条,含参考答案/标注,刻意覆盖难例;标注难度分层。附标注规范。
D3
评测脚本 —— 跑成对比较 + LLM 评委 + 人工校准抽样 + bootstrap 置信区间;输出能力矩阵(含成本/时延)。
D4
能力矩阵报告 —— 模型 × 任务类 的质量/通过率/成本/时延,带置信区间;哪些差距是真的、哪些是噪声要写清。
D5
路由决策表 v1 —— 任务类 → 默认模型 + 升级触发条件 + 硬门槛校验;可直接查表使用。
D6
路由日志方案 —— 线上每次决策记录什么字段,供后续升级 kNN/分类器。
D7
结题汇报 —— 方法、结论、省成本测算、下一步建议。

9验收标准

  • 任务分类体系覆盖了 ≥ 90% 的真实用例,边界清晰,导师评审通过。
  • Golden set 每类样本充足且含难例;标注规范可复现(两人标注一致性达标)。
  • 评测可一键复跑;LLM 评委与人工一致性(κ)达到约定阈值。
  • 能力矩阵的结论带置信区间,明确区分"真差距"与"噪声"。
  • 路由决策表能对一批新任务给出合理选择,并通过硬门槛校验。
  • 给出量化的省成本测算(对比"全用强模型" vs "按路由")。

10待确认事项 + 避坑

动手前请与导师确认(会改变方案细节):
  1. 候选模型是谁?是公司基座模型的不同尺寸/版本之间选,还是基座 vs 外部模型(GPT/Claude/Gemini)混合?——影响成本维度与合规约束。
  2. 路由是离线还是在线?是"每条业务线提前定死用哪个模型"(那做完评测就够了),还是"每个请求实时路由"(才需要完整路由器)?
  3. 合规/数据安全边界:用例数据能否出内网?评委模型能否用外部 API?
  4. 各任务类的延迟/成本预算红线。
常见坑:
  • 把能力压成一个总分 → 丢掉路由最需要的"按任务变化"。
  • 用绝对打分而非成对比较 → 小数据下排名不稳。
  • 盲信 LLM 评委、不做人工校准 → 评测本身就错。
  • 用整体 accuracy 评分类 → 漏掉高代价长尾类(投诉/合规)。要看召回门槛。
  • 小数据上训复杂路由器 → 过拟合。先用规则表。
  • 为噪声做决策 → 差距没过置信区间就别区分,选便宜的。
